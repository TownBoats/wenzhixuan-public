# 维护待办

## 说明

本文基于当前代码库中与 AI 请求、流式解析、消息状态、模型配置、欢迎页提示、重试机制相关的实现整理。

优先级说明：

- `P0`：需要优先处理，已经影响核心可用性、数据一致性或安全性
- `P1`：建议紧接着处理，关系到能力扩展、交互正确性和后续维护成本
- `P2`：建议在主链路稳定后处理，属于体验、健壮性和工程优化

## 改造约束

本清单默认遵循“**不能影响现有功能**”的原则，具体约束如下：

- 不做一次性推倒重写
- 优先增加兼容层，而不是直接删除旧实现
- 先保证现有发送、流式展示、历史记录、答案卡链路可继续工作
- 新结构先以“旁路接入”或“渐进切换”方式上线
- 每一步改造都应具备可回退性
- 若某项改动会影响当前用户路径，必须拆成更小步骤执行

因此，下文中的“合并”“统一”“重写”都不是指一次性替换，而是指：

- 先补一层适配
- 再双写或影子运行
- 验证稳定后再切主
- 最后清理旧代码

## 总体判断

当前主链路可概括为：

`ChatPage -> useAgentsConfig -> AgentModel -> useStreaming / StreamingParser -> ChatWindow / MessageBubble`

现阶段的主要问题不是单点组件瑕疵，而是以下几类基础设计缺口叠加：

- 会话状态分成 `messages` 和 `uiMessages` 两套，已经出现漂移
- 请求失败、自动重试、UI 重试三套语义没有统一
- 模型配置只支持最基础字段，缺少高级参数与能力抽象
- “思考 / 非思考 / 最终回答”没有统一的数据结构与渲染模型
- 首页“需要配置模型”的提示没有基于真实配置状态判断

## 建议执行顺序

### 第一阶段：先做兼容式止血，不动主渲染模式

1. 给消息补稳定 ID 和映射关系
2. 修复请求失败与重试机制
3. 改成基于消息 ID 或映射关系的重试
4. 修复失败态消息显示与原位重试
5. 删除硬编码 API Key
6. 修复 `deselect` 和 quick prompt 等直接运行问题
7. 增加回归验证清单

### 第二阶段：在兼容层下逐步补齐模型与思考能力

1. 扩展模型配置结构，默认值保持当前行为
2. 建立 provider / capability 适配层
3. 定义 reasoning / response / question 的统一段落结构
4. 通过适配器把新段落结构回落为旧 UI 可渲染格式
5. 完成“思考内容”的 UI 展示、折叠、隐藏策略
6. 补齐选项 AI 的错误处理

### 第三阶段：整理体验与工程质量

1. 修复欢迎页配置提示判断
2. 清理语言切换与提示词同步逻辑
3. 在确认新结构稳定后，再清理冗余状态和旧实现
4. 补测试
5. 处理包体与模块拆分

## 建议验收标准

### P0 验收

- 普通发送失败后，界面出现失败态消息，且可直接重试
- assistant 重试与 user 重试都不会错位
- 历史记录恢复后，重试仍然命中正确消息
- 即使仍保留两套消息数组，也不再依赖它们的相同索引
- 仓库内不再存在前端默认 API Key
- 修复后发送、流式显示、答案卡、历史记录功能仍与当前表现一致

### P1 验收

- 设置页可配置高级模型参数
- 不同模型可按能力开关“思考模式”
- UI 可明确区分思考内容与最终回答
- 选项 AI 失败时问题卡有明确错误状态和恢复路径
- 首页仅在真实未配置时提示设置
- 新能力关闭时，现有交互与当前版本保持一致

### P2 验收

- 语言切换后提示词来源单一、行为可预测
- 关键链路具备最小自动化测试
- 代码中明显失效的状态和分支被清理

## 备注

如果只做一轮最小修复，建议至少覆盖以下范围：

- `AgentModel`
- `ChatPage`
- `useQuestionFlow`
- `ModelConfigPanel`
- `WelcomeScreen`

这五处处理完，当前最影响使用的问题基本能落地解决。

如果严格要求“不能影响现有功能”，则建议首轮只做：

- 修复重试底层 bug
- 修复索引错位
- 修复运行时错误
- 删除硬编码 Key
- 修复欢迎页误提示

先不在首轮引入大规模状态模型切换。等现有功能稳定后，再进入兼容式结构升级。
